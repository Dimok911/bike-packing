# Логика копирования шаблонов

## Главное правило

Фронт не показывает шаблон как существующий, пока сервер не подтвердил его существование.

Серверный список опубликованных шаблонов является единственным источником того, какие шаблоны видны в админке и без логина. Этот список собирается из реальных строк `bike_packing_lists`: у каждой строки есть `id`, `title`, `language`, `source_type`, `visibility` и `payload`. Локальная копия на фронте нужна только для редактирования уже подтвержденной серверной записи.

Demo и референс-шаблоны относятся к одному публичному каталогу шаблонов. Разница только в роли: demo показывает возможности программы и может стать стартовой укладкой без логина, а референс-шаблоны являются реальными опубликованными укладками людей, на которые можно ориентироваться. В старом коде роль референс-шаблона может называться `shared`, но в интерфейсе это слово не показывается: пользователь видит просто `Шаблон: Название (RU/EN)`.

Отличие от пользовательской укладки: копия шаблона становится видимой только после ответа сервера, потому что это опубликованная запись для всех. Пользовательская укладка может появиться локально сразу, потому что она принадлежит текущему пользователю и потом синхронизируется как личные данные.

## Копирование шаблона

1. Пользователь нажимает копирование шаблона.
2. Фронт собирает новую копию локально, но не считает ее видимой опубликованной записью.
3. Фронт сразу отправляет новую копию на сервер.
4. Сервер создает новую опубликованную запись шаблона и сохраняет ее данные.
5. Фронт заново получает серверный список опубликованных шаблонов.
6. Если новая запись есть в серверном списке, фронт открывает новую копию для редактирования и показывает ее в админском списке.
7. Если сервер не подтвердил новую запись, фронт убирает локальную заготовку и не показывает ее в списке.

## Оффлайн

1. Без связи с сервером копирование опубликованного шаблона не завершается как успешное.
2. Локальная заготовка не становится видимой строкой в админском списке.
3. Когда сервер недоступен или не подтвердил запись, фронт убирает заготовку и показывает ошибку.
4. В оффлайне можно продолжать работать с уже загруженными шаблонами, но нельзя создать новую опубликованную копию шаблона.
5. В принудительном оффлайн-режиме или при отсутствии интернета published-шаблоны заблокированы: их нельзя открыть, редактировать, копировать или удалять как серверные шаблоны.
6. Личные пользовательские укладки в оффлайне остаются доступными отдельно от published-шаблонов.

## Смена языка шаблона

1. Язык published-шаблона является серверной меткой каталога.
2. При смене языка фронт меняет только серверную метку, а не содержимое шаблона.
3. Публичный показ без логина раскладывает шаблоны по языкам именно по этой серверной метке.
4. Если payload шаблона внутри еще содержит старый язык, это не должно переигрывать серверную метку каталога.

## Demo и референс-шаблоны при выборе языка

Demo и референс-шаблоны работают как один тип публичных шаблонов: это read-only источники, которые приходят из общего каталога metadata и показываются в списке шаблонов выбранного языка. Разница только в роли: demo является стартовым шаблоном и демонстрацией возможностей программы, а референс-шаблон является опубликованной укладкой реального пользователя.

1. В каталоге у demo и референс-шаблонов есть общие поля: роль, id, название, язык, время обновления и признак серверного подтверждения.
2. В выбранном языке список показывает только шаблоны этого же языка.
3. Если референс-шаблона нет на сервере, фронт не подставляет встроенную или локально оставшуюся версию.
4. Если demo-шаблона нет на сервере для выбранного языка, показывается пустая изолированная заготовка, а не старая встроенная демо-версия.
5. Если demo-шаблонов в одном языке несколько, стартовым считается первый в списке этого языка.
6. Demo может использоваться как стартовая заготовка для первой личной укладки без логина. Это не меняет правило каталога: сама demo все равно остается публичным шаблоном выбранного языка.

## Смена языка при просмотре

При смене языка текущий выбор должен оставаться в той же смысловой роли, но перейти на шаблон выбранного языка.

1. Если открыт demo-шаблон, после смены языка открывается demo-шаблон нового языка.
2. Если открыт референс-шаблон, фронт ищет референс-шаблон нового языка с той же логической парой.
3. Сначала пара ищется по общей семье шаблона, затем по одинаковому названию, затем по одинаковому порядковому номеру в названии.
4. Если точной пары нет, выбирается ближайший шаблон в списке нового языка.
5. Если в новом языке нет ни одного референс-шаблона, фронт переходит на первый demo-шаблон нового языка.
6. Нельзя показывать метку нового языка поверх содержимого шаблона старого языка. Выбранный шаблон и его содержимое должны принадлежать одному языковому списку.

## Язык в админке

Админка назначает шаблону язык как свойство каталога. Это не автоматический перевод и не копирование содержимого.

1. Demo редактируется отдельно для каждого языка: RU demo и EN demo являются разными опубликованными источниками.
2. Копирование demo в админке создает новый demo-шаблон с новым публичным id и следующим порядковым названием по умолчанию.
3. Референс-шаблон при назначении языка попадает в список этого языка.
4. Смена языка шаблона в админке переносит шаблон между языковыми списками, но не меняет вещи, сумки, названия и тексты внутри.
5. Название и язык сохраняются в серверную metadata той же строки шаблона, а содержимое, фото и раскладка сохраняются в payload этой же строки.
6. Для полноценной пары RU/EN нужны две опубликованные версии: по одной на каждый язык.
7. Публичный режим и безлогин смотрят на серверный язык шаблона, а не на старые локальные следы.

## Безлогин и автоматическая demo-укладка

При входе без логина demo-шаблон является стартовой заготовкой для локальной гостевой укладки. Это отдельное правило только для demo; референс-шаблоны сами локальные укладки не создают.

1. Если пользователь без логина заходит в обычный режим приложения через demo-шаблон, фронт автоматически создает локальную гостевую укладку из первого demo выбранного языка.
2. Имя этой локальной укладки при создании должно совпадать с названием demo-шаблона из metadata/catalog, без автоматических суффиксов `2`, `3`, `4`.
3. Повторный вход без логина не должен создавать новую demo-укладку. Фронт должен открыть уже созданную гостевую demo-копию или обновить ее служебное имя, если она еще не редактировалась пользователем.
4. Новая дополнительная demo-копия создается только по явному действию пользователя, например через кнопку взять demo как новую укладку.
5. Если server demo для выбранного языка отсутствует, встроенная старая demo-версия не подставляется. В этом случае стартовая заготовка пустая, но правило идемпотентности остается тем же: не плодить новые локальные укладки при каждом входе.
6. Смена языка у public-шаблона меняет источник demo на demo выбранного языка. Уже созданная локальная гостевая укладка остается пользовательской локальной укладкой и не должна превращаться в public-шаблон.

## Metadata

### 2026-05-26 public template language model

Demo and reference templates use one authoritative metadata model for language. The server reads `language` from `bike_packing_lists.language` for both kinds of public templates.

Legacy ids such as `public-demo-state-en` remain technical row ids only. They do not decide the language anymore. The old demo `sharedLayoutsIndex` is also not a catalog source for reference templates anymore: if a reference template has no real public list row with `language`, it is not shown.

Deployment order: apply `bikepacking-api/docs/bike-packing-public-template-language-migration.sql`, deploy API `2026-05-26.public-template-language-column-v1`, then deploy frontend `v816`.

Metadata - это короткое описание записи для каталога, а не сама укладка. В metadata хранятся признаки, по которым фронт может быстро понять, что это за шаблон и где его показывать: роль demo/reference, название, язык, видимость, тип источника, id, время обновления и похожие служебные поля.

В `bike_packing_lists` название шаблона хранится в столбце `title`, язык хранится в столбце `language`. Именно эти поля должны меняться при редактировании названия или языка в админке. Содержимое шаблона хранится отдельно в `payload`.

Metadata не является содержимым шаблона. Вещи, сумки, группы, веса, фотографии и раскладка живут в payload. Metadata нужна, чтобы построить списки и выбрать язык без загрузки тяжелого payload. Если metadata говорит, что шаблон RU, публичный каталог считает его RU, даже если внутри payload остались старые подписи или старое значение языка.

Единый каталог metadata от сервера должен отдавать demo и референс-шаблоны вместе. Если записи нет на сервере, фронт не должен восстанавливать ее из старого встроенного набора или локального runtime-следа. Логика выбора остается общей: сначала metadata/язык/роль, потом загрузка payload.

## Удаление шаблона

1. Пользователь удаляет опубликованный шаблон.
2. Фронт отправляет удаление на сервер.
3. Сервер удаляет опубликованную запись шаблона.
4. Фронт убирает все локальные следы этой записи.
5. Фронт заново опирается только на серверный список опубликованных шаблонов.
6. После удаления фронт открывает следующий шаблон по порядку.
7. Если следующего референс-шаблона нет, фронт остается в админской зоне шаблонов и открывает demo-шаблон, а не пользовательскую укладку.

## Запрещено

- Показывать template строку только потому, что она осталась в локальном state.
- Показывать template строку только потому, что она осталась в runtime-каталоге.
- Считать локальный draft опубликованным шаблоном без подтверждения сервера.
- После удаления опубликованного шаблона переключаться в пользовательские укладки.
