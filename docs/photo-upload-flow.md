# Логика загрузки фото

Документ описывает целевую модель загрузки фото в bike-packing. Это контракт для переписывания модуля загрузки, UI модалки, синхронизации и critical-тестов.

## Коротко

Загрузка фото должна ощущаться для пользователя как один спокойный процесс: выбрал фото, увидел preview, при сохранении/синхронизации фото либо загрузилось, либо понятно осталось ждать повторной загрузки, либо показало явную ошибку.

Главное правило: отправка байтов из браузера еще не означает, что фото сохранено на сервере. Приложение не должно показывать "готово" до ответа сервера с настоящей ссылкой на фото.

В модалке редактирования при настоящей отправке фото должен быть один круг progress с процентами. В интерфейсе не должно быть второго круга прогресса, скачка `100 -> 0`, вечного "Ждем загрузки" или блокировки кнопки "Сохранить" из-за временного spinner. Кнопка "Сохранить" должна оставаться рабочей и зависеть от реальных изменений в форме.

Локальное фото, которое есть только на этом телефоне или компьютере, нельзя отправлять в общий server state как будто оно уже доступно всем устройствам. Иначе на другом устройстве появится пустой квадрат или бесконечная загрузка.

Вся сложная логика загрузки должна быть собрана в отдельном модуле. `app.js` должен только связать UI, endpoint, callbacks и сохранение состояния.

## Цель

Загрузка фото должна быть одним понятным потоком с одним источником правды на уровне записи фото. UI не должен создавать отдельные временные состояния, которые живут отдельно от самой записи фото.

`app.js` не должен содержать бизнес-логику загрузки. Его роль:

- определить endpoint и list/layout context;
- передать entity, photo и зависимости в модуль загрузки;
- вызвать render/save callbacks;
- показать toast/status при необходимости.

Вся логика переходов состояния, retry/recovery, обработки XHR progress и серверного результата должна жить в профильных модулях, прежде всего в `src/sync/...`.

## Состояния фото

Фото хранит состояние в самой записи:

- `pending` - локальное фото уже подготовлено и сохранено в IndexedDB, но серверный файл еще не подтвержден;
- `uploading` - прямо сейчас идет конкретная попытка upload для этой записи;
- `synced` - сервер подтвердил фото и у записи есть реальный `url` или `thumbUrl`;
- `error` - upload завершился явной ошибкой, которую нельзя безопасно оставить как queued retry;
- `missing-local-file` - запись ссылается на локальный файл, но blob не найден.

`synced` можно ставить только после успешного ответа сервера. Наличие `xhr.upload.onload` или `progress = 100` не является подтверждением `synced`.

## Progress

XHR upload progress означает только отправку байтов из браузера на сервер.

Правила:

- `xhr.upload.onprogress` может обновлять видимый прогресс от `0` до `99`;
- `xhr.upload.onload` означает только "браузер отправил байты";
- `xhr.upload.onload` не ставит `100`, не ставит `synced`, не очищает фото как готовое;
- `100` допустим только после успешного `xhr.onload`/server response с данными фото;
- после server response модуль применяет `url/thumbUrl`, ставит `synced` и очищает transient progress;
- UI не должен запускать второй progress cycle после окончания отправки байтов.

Если сервер еще обрабатывает фото после того, как браузер отправил bytes, первый progress остается активным в состоянии ожидания server response. Нельзя показывать второй кружок с `0`.

## Серверный результат

Успешный ответ сервера обязан дать достаточно данных, чтобы запись стала серверной:

- `id` или fallback photo id;
- `url` и/или `thumbUrl`;
- `listId` или возможность вычислить list-scoped URL;
- `updatedAt`, если сервер его возвращает.

После успешного ответа:

1. Найти актуальную запись фото внутри entity.
2. Применить серверные поля.
3. Сохранить локальный `localId`, если он нужен как offline fallback.
4. Поставить `status = "synced"`.
5. Очистить `error`, `_copyToCurrentList`, `copyToCurrentList`.
6. Очистить transient progress.
7. Обновить `updatedAt` entity.
8. Запланировать render/save через callbacks, а не через внутренний UI-state.

## Ошибки и retry

Retry должен быть явным и не должен создавать второй UI progress cycle.

Временные сетевые ошибки, timeout и stalled upload могут вернуть фото в `pending`, если:

- локальный blob есть;
- это приватный пользовательский endpoint, а не admin/template endpoint;
- retry разрешен вызывающим сценарием.

Для dialog draft upload retry по умолчанию не должен сразу запускать вторую отправку. Достаточно вернуть фото в `pending` или показать recoverable состояние. Пользовательская save-кнопка при этом не должна превращаться в вечное ожидание.

Если есть подозрение, что сервер уже сохранил фото, но клиент не дождался ответа, модуль может попытаться resolve by content hash. Это recovery, а не второй progress cycle.

Если recovery нашел фото:

- применить серверный результат;
- поставить `synced`;
- очистить progress.

Если recovery не нашел фото, но ошибка временная:

- вернуть `pending`;
- очистить progress;
- оставить локальный blob доступным для следующего upload.

Если ошибка невосстановимая:

- поставить `error`;
- записать понятный `error`;
- очистить progress.

## Sync payload

Remote sync payload не должен содержать local-only фото.

Запрещено сериализовать фото, если у него нет безопасного серверного `url` или `thumbUrl`.

Это защищает сценарий:

- на телефоне фото видно из IndexedDB/local blob;
- на другом устройстве этого blob нет;
- если local-only pending запись утечет в remote state, второе устройство увидит пустой квадрат или бесконечный spinner.

Для sync допускаются только серверные фото:

- `synced` с `url/thumbUrl`;
- pending с `url/thumbUrl`, если это устаревшая или переходная запись, но UI не должен считать ее "Ждем загрузки".

## Pending с серверными URL

Если у фото `status = "pending"`, но есть серверный `url` или `thumbUrl`, UI не показывает "Ждем загрузки".

Такую запись нужно воспринимать как usable remote photo для отображения. Нормализация или следующий save/upload flow должны привести ее к `synced`, но UI не должен пугать пользователя критикой.

## Диалог редактирования

Модалка редактирования вещи или сумки работает с photo draft.

Правила:

- выбор файла добавляет фото в draft и сразу показывает preview из локального blob;
- отмена file picker не закрывает модалку;
- клик по свободному месту после выбора фото не закрывает модалку;
- save-кнопка зависит от имени и реальных изменений draft/form;
- save-кнопка не зависит от временного upload spinner;
- save-кнопка не меняет текст на "Фото загружается";
- preview не должен мигать из-за промежуточного upload-state;
- локальный `pending` draft после выбора фото не является активной загрузкой и не рисует progress-круг;
- локальный `pending` draft после выбора фото не пишет в диалоге "Ждем загрузки";
- upload progress в модалке рисуется только для реального `uploading`, а не из отдельного dialog flag;
- при старте реального upload модалка сразу показывает progress-круг с `0%` уже в первом preview-render, не дожидаясь первого XHR progress event;
- progress в модалке обновляется точечно внутри уже отрисованной галереи, без пересоздания preview;
- финиш upload не пересоздает preview и не сносит progress overlay полным render;
- после сохранения карточка вещи/сумки рисует тот же progress-круг, если upload еще реально идет;
- пакет из нескольких фото показывает на галерее `Фото N из M`; этот номер и progress-круг текущего файла не сбрасываются при переходе из модалки в сохранённую карточку;
- отдельный fixed/header progress для фото не используется: источник отображения прогресса один и тот же в модалке и на основной странице;
- после успешного server response карточка сразу перестает показывать progress и текст загрузки.

Временные fallback-слои в диалоге запрещены:

- `dialogPhotoUploadInProgress`;
- `markDialogPhotosUploading`;
- `markUnresolvedDialogUploadsFailed`;
- `updateDialogPhotoUploadProgress`;
- `showUploadState`;
- отдельный overlay progress поверх галереи, если уже есть progress у фото;
- любые save button states вида "Фото загружается".

## Touch и галерея

Галерея фото должна поддерживать горизонтальный свайп и не ломать вертикальный scroll страницы или модалки.

Правила:

- не использовать `touch-action: pan-x`, если это блокирует вертикальный scroll;
- допустимо `touch-action: pan-x pan-y`, если вертикальный scroll сохраняется;
- свайп по фото не должен создавать press-effect на кнопке/слайде;
- вертикальные жесты не должны превращаться в горизонтальное гуляние галереи;
- click после swipe не должен открывать lightbox случайно.

## Полноразмерный просмотр

- lightbox открывается сразу на уже показанном preview и не ждёт сеть перед появлением;
- полная версия загружается и декодируется в отдельном `img`, после чего атомарно заменяет preview;
- неготовая или повреждённая полная версия не должна убирать уже видимое preview;
- индикатор полной версии появляется только после короткой задержки, чтобы быстрый cache hit не мигал;
- если полной версии нет, пользователь видит двуязычный статус preview;
- боковые зоны навигации занимают всю высоту экрана, а видимый размер стрелок не увеличивается.
- remote-оригинал из IndexedDB используется только после явной проверки
  `fullBlobVerified` и при совпадении `sourceSignature`;
- миниатюра не может незаметно занять место оригинала; если офлайн доступен
  только сохранённый preview, lightbox сообщает об этом отдельно;
- проверенная полная версия остаётся доступной после повторного рендера карточек
  и не заменяется миниатюрой при перелистывании.
- оригиналы до 1 мегапикселя не увеличиваются автоматически выше естественного
  размера; это ограничение пересчитывается после декодирования каждого фото и
  при изменении viewport, а ручной zoom остаётся доступен.

## iPhone и iCloud

Выбранное фото из iCloud должно быть материализовано до resize/upload.

Правила:

- сначала прочитать файл/blob полностью;
- системный iOS picker может закрыться раньше, чем iCloud фактически догрузил выбранное фото; приложение обязано дождаться материализованного файла, а не создавать draft из полупустого/ленивого `File`;
- если iCloud read завис, завершить с понятной ошибкой;
- не создавать upload draft, если файл не материализован;
- preview в модалке можно менять только после готового локального blob/thumb; не должно быть трех миганий фото из-за промежуточных стадий materialize/resize/upload;
- progress upload не должен стартовать, пока выбранный iCloud-файл не прочитан, уменьшен и записан в локальный photo cache;
- если выбранное iCloud-фото не удалось материализовать, в draft не появляется pending-запись, а пользователь видит понятную ошибку и может выбрать фото заново;
- stalled upload должен завершаться error/recovery, а не вечным `uploading`;
- при stream stall нужно использовать свежий blob/slice для следующей попытки.

## Кнопка "Сделать главным"

Контракт кнопки не должен ломаться при переписывании фото-модуля:

- 0 фото - кнопки нет;
- 1 фото - кнопки нет или disabled согласно текущему UI-контракту;
- 2+ фото - кнопка видна;
- активное фото первое - disabled, текст "Фото уже главное";
- активное фото не первое - enabled, текст "Сделать главным".

## Целевые модули

Рекомендуемая структура:

- `src/sync/photo-upload-flow.js` - основной upload state machine;
- `src/sync/api-client.js` - только transport: XHR, timeout, stalled upload, progress events;
- `src/sync/photos.js` - операции над photo record: normalize, apply synced result, pending retry, content hash resolve;
- `src/sync/serialize.js` - pruning local-only photos from sync payload;
- `src/ui/photo-gallery.js` - отображение photo state и progress;
- `src/app/app-tail-controllers.js` - dialog draft orchestration without upload business rules;
- `app.js` - thin glue for endpoints, callbacks, render/save.

## Critical-тесты

Минимальные контракты:

- `xhr.upload.onload` не делает фото `synced` и не дает финальный `100`;
- progress до server response ограничен `99`;
- server response переводит фото в `synced` и очищает progress;
- dialog upload не создает второй progress cycle;
- local-only `pending/uploading` не попадает в sync payload;
- pending с server URL не показывает "Ждем загрузки";
- save button не получает текст "Фото загружается";
- file picker cancel не закрывает модалку;
- клик по свободному месту после выбора фото не закрывает модалку;
- iOS file picker dismiss не закрывает модалку редактирования вещи/сумки;
- iCloud file materialization завершается до появления draft/preview/upload;
- неуспешная materialization не создает pending-фото и не запускает вечный spinner;
- локальный dialog draft в статусе `pending` не показывает progress-круг до старта настоящего upload;
- старт настоящего dialog upload сразу переводит фото в `uploading` и показывает `0%`;
- локальный dialog draft в статусе `pending` не показывает текст "Ждем загрузки" над save-кнопкой;
- dialog upload progress обновляет существующий progress-круг, а не пересоздает preview;
- успешный dialog upload очищает/обновляет progress точечно, без полного rerender фото;
- сохраненная карточка вещи/сумки показывает progress-круг только для активного `uploading`;
- после `synced` сохраненная карточка вещи/сумки не показывает progress-круг и текст "Ждем загрузки"/"Фото загружается";
- dialog preview не пересоздает галерею на каждый upload progress tick;
- touch-action сохраняет вертикальный scroll;
- primary photo button сохраняет свой контракт.

## Проверки после правки

Если меняется runtime/UI/cache:

- bump `src/config/constants.js` `APP_VERSION`;
- bump query string в root `index.html` для `styles.css` и `app.js`;
- bump `CACHE_NAME` и `ASSETS` в root `sw.js`.

Запустить:

- `npm.cmd run check`;
- `npm.cmd run test:critical`;
- `npm.cmd run build`.

Автоматической проверки in-app browser в текущем окружении нет. Жесты, iPhone/iCloud и реальное поведение file picker нужно проверить вручную на устройстве.
