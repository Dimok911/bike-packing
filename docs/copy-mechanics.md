# Механики копирования

Документ фиксирует ожидаемое поведение копирования сумок, вложенных сумок и вещей. Если меняется UI или контракт копирования, сначала обновить этот файл, затем код и регрессионные проверки.

## Термины

- **Пользовательская укладка** - приватная укладка гостя или авторизованного пользователя.
- **Demo-укладка** - основной опубликованный шаблон для выбранного языка. Без
  логина demo может стать стартовой локальной пользовательской укладкой, но
  опубликованная demo-укладка не становится личной укладкой пользователя сама
  по себе.
- **Shared-укладка** - дополнительная опубликованная укладка-шаблон из
  публичного каталога. Обычные пользователи видят ее как read-only источник для
  копирования, админ может открыть ее как редактируемый draft.
- **Шаблон** - собирательный термин для опубликованной demo-укладки или
  shared-укладки. В админском режиме шаблон может быть открыт как
  редактируемый draft.
- **Текущая укладка** - укладка, из которой пользователь начал действие. При копировании из шаблона текущий шаблон не должен быть целевой укладкой.
- **Место назначения** - конкретный контейнер внутри целевой укладки или корень целевой укладки.
- **Личные данные пользователя** - все пользовательские укладки одного гостя
  или аккаунта. У них общий каталог вещей, общий каталог сумок, общие
  справочники и общие фото.
- **Собственные данные шаблона** - вещи, сумки, справочники, размещение и фото,
  которые принадлежат только конкретной demo/shared укладке. RU и EN версии
  одного шаблона считаются разными шаблонами со своими собственными данными.

## Гость без логина

Гость работает с локальными пользовательскими укладками. Если пользовательской укладки еще нет, при копировании из шаблона создается локальная укладка.

| Что копируем | В пределах текущей пользовательской укладки | В другую пользовательскую укладку | Из шаблона |
| --- | --- | --- | --- |
| Верхнеуровневая сумка | Открывается root-level интерфейс как при перестановке сумок: горизонтальные колонки укладки и слоты между верхнеуровневыми сумками. | Сначала выбирается целевая пользовательская укладка, затем тот же root-level интерфейс в ней. | Открывается тот же root-level интерфейс. В списке целей только пользовательские укладки гостя, кроме текущей, если она уже содержит эту сумку; при отсутствии целей создается новая локальная укладка. |
| Вложенная сумка | Открывается выбор места внутри текущей укладки. Нельзя выбрать саму сумку или ее потомков. | Открывается тот же выбор места в целевой укладке. | Открывается тот же выбор места в пользовательской целевой укладке. |
| Вещь | Открывается выбор сумки/места внутри текущей укладки. Размещение использует существующий `id`; настоящий дубль создается только специальной кнопкой дублирования на карточке. | Открывается тот же выбор сумки/места в целевой укладке. Размещение использует существующий `id`. | Открывается тот же выбор сумки/места в пользовательской целевой укладке. |

Фото из шаблона кэшируются локально как fallback для последующей синхронизации после входа.

## Обычный пользователь

Обычный пользователь работает с приватными серверными укладками. Копирование из шаблона не должно менять сам шаблон.

| Что копируем | В пределах текущей пользовательской укладки | В другую пользовательскую укладку | Из шаблона |
| --- | --- | --- | --- |
| Верхнеуровневая сумка | Открывается root-level интерфейс как при перестановке сумок: горизонтальные колонки укладки и слоты между верхнеуровневыми сумками. | Открывается тот же root-level интерфейс в выбранной пользовательской укладке. | Открывается тот же root-level интерфейс. В списке целей пользовательские укладки, кроме текущей, если она уже содержит эту сумку. |
| Вложенная сумка | Открывается выбор места внутри текущей укладки; потомки исходной сумки недоступны как цель. | Открывается тот же выбор места в другой пользовательской укладке. | Открывается тот же выбор места в пользовательской целевой укладке. |
| Вещь | Открывается выбор сумки/места. Размещение использует существующий `id`; настоящий дубль создается только специальной кнопкой дублирования на карточке. | Открывается выбор сумки/места в другой укладке. Размещение использует существующий `id`. | Открывается выбор сумки/места в пользовательской целевой укладке. |

Если исходный объект уже является копией из public/template, при копировании сохраняется защита от перезаписи публичного происхождения и проверяются дубли.

## Проверка дублей внутри личных данных пользователя

Внутри пользовательских укладок одного пользователя копирование в другую укладку
работает как размещение существующих записей каталога, а не как создание новых
записей во вкладках "Вещи" и "Сумки".

- Вещь, которой еще нет в целевой укладке, добавляется в выбранную сумку по
  существующему `id`.
- Если эта вещь уже есть в целевой укладке, приложение должно спросить
  пользователя. Только явное действие "Дублировать" создает новую запись вещи в
  каталоге; отмена/пропуск не меняет каталог.
- Верхнеуровневая сумка, которой еще нет в целевой укладке, добавляется в
  целевую укладку по существующему `id`. Название не получает суффикс "копия",
  а вкладка "Сумки" не раздувается новой записью.
- Если такая сумка или часть ее дерева уже есть в целевой укладке, сначала
  выполняется проверка дублей. Сценарий "Только недостающие" восстанавливает
  отсутствующие вложенные контейнеры/пакеты и вещи по существующим `id` без
  создания новых записей каталога и без дублирования уже существующих элементов.
  Сценарий "Дублировать" создает настоящий дубль только после явного выбора
  пользователя.
- Специальные кнопки дублирования на карточках во вкладках "Вещи" и "Сумки" -
  отдельный сценарий. Они действительно создают новую запись каталога и могут
  добавить суффикс "копия".

## Модель копирования фото

Фото не являются самостоятельным глобальным справочником. Их область хранения
определяется тем, где находится укладка: в личных данных пользователя или в
собственных данных конкретного шаблона.

- **Пользовательские укладки одного пользователя** живут в одних личных данных.
  При копировании вещей, сумок и содержимого сумок между такими
  укладками фото физически не копируются: целевая укладка получает ссылку на
  тот же `id` вещи или сумки, а фото остаются ссылками на те же серверные файлы
  этого же списка. Если пользователь явно создает дубль специальной кнопкой на
  карточке, тогда появляется новая запись каталога.
- **Вложенный пакет/список внутри сумки** является частью конкретной укладки, а
  не общей root-level сумкой каталога. При переносе такого пакета в другую
  пользовательскую укладку создается новая вложенная структура целевой укладки,
  чтобы она жила независимо от исходной. При этом фото внутри тех же личных
  данных не отправляются через `/photos/copy`: если у фото уже есть
  серверные `url`/`thumbUrl`, новая вложенная структура сохраняет эти ссылки как
  `synced`; если фото еще только локальное, копируется локальный blob в
  IndexedDB под новым `localId`.
- **Шаблон** хранит собственные данные отдельно от личных данных пользователя.
  Каждый demo/shared/template list,
  включая отдельные RU/EN шаблоны, хранит собственный каталог и собственные
  файлы фото.
- Если копирование пересекает границу пользовательская укладка -> шаблон,
  шаблон -> пользовательская укладка или шаблон -> другой шаблон, фото должны
  быть физически скопированы в целевой list.
- Маркер `_copyToCurrentList` допустим только для перехода через такую границу.
  Для копий внутри приватных пользовательских укладок этот маркер ставить
  нельзя, иначе приложение начнет дергать `/photos/copy` для фото, которые уже
  доступны в тех же личных данных пользователя.
- Явное private -> private дублирование через copy-to-layout может создать новую
  запись вещи или контейнера только после подтверждения пользователя, но оно все
  равно остается внутри тех же личных данных: remote photo id не меняются и
  `/photos/copy` не вызывается.

## Админ

Админ может копировать в пользовательские укладки и в другие редактируемые draft-шаблоны. Текущий draft-шаблон не должен быть целевой укладкой при копировании из него.

| Что копируем | В пределах текущей пользовательской укладки | В другую пользовательскую укладку | Из шаблона |
| --- | --- | --- | --- |
| Верхнеуровневая сумка | Открывается root-level интерфейс как при перестановке сумок: горизонтальные колонки укладки и слоты между верхнеуровневыми сумками. | Открывается тот же root-level интерфейс в выбранной пользовательской укладке. | Открывается тот же root-level интерфейс. В списке целей пользовательские укладки и другие draft-шаблоны, кроме текущего draft-шаблона. |
| Вложенная сумка | Открывается выбор места; нельзя выбрать саму сумку или ее потомков. | Открывается тот же выбор места в другой пользовательской укладке или draft-шаблоне. | Открывается тот же выбор места в пользовательской укладке или другом draft-шаблоне. |
| Вещь | Открывается выбор сумки/места. В пользовательской укладке размещение использует существующий `id`; дубль создается только специальной кнопкой на карточке. | Открывается выбор сумки/места в другой пользовательской укладке или draft-шаблоне. Для пользовательской цели используется существующий `id`, для draft-шаблона создается запись целевого шаблона. | Открывается выбор сумки/места в пользовательской укладке или другом draft-шаблоне. |

При копировании в draft-шаблон новые записи маркируются как принадлежащие целевому public layout. При копировании в приватную укладку public/template origin очищается или сохраняется только как безопасная ссылка для дедупликации.

## Public/template metadata

Серверный список опубликованных demo/shared/template записей является
авторитетным источником того, какие шаблоны видны в админке и без логина.
Фронт не должен показывать template-строку только потому, что она осталась в
локальном state или runtime-каталоге.

- Копия опубликованного шаблона считается видимой только после подтверждения
  сервера и повторного получения серверного каталога.
- Demo и референс-шаблоны используют одну metadata-модель: роль, `id`,
  название, язык, видимость, тип источника и время обновления живут в
  серверной строке `bike_packing_lists`.
- Язык шаблона хранится как metadata (`bike_packing_lists.language`) и не
  переводит содержимое payload.
- RU и EN версии одного шаблона - разные опубликованные записи с разными
  собственными данными. Для полноценной пары нужны две серверные версии.
- Если серверной записи шаблона нет, фронт не восстанавливает ее из старого
  встроенного набора или локальных следов.
- При смене языка публичного шаблона фронт выбирает шаблон из списка нового
  языка, а не подменяет metadata поверх payload старого языка.
- Без логина demo может стать стартовой локальной пользовательской укладкой.
  Повторный вход не должен плодить новые demo-копии без явного действия
  пользователя.
- Удаление опубликованного шаблона сначала подтверждается сервером. После
  удаления фронт очищает локальные следы этой записи и остается в зоне
  шаблонов/demo, а не переключается в пользовательские укладки.

## UI-контракт

- Индивидуальная кнопка копирования сумки или вещи из шаблона открывает picker места назначения.
- Picker в режиме копирования показывает выбор целевой укладки, если доступно больше одной цели.
- Для копирования верхнеуровневого контейнера используется root-level доска со слотами между верхнеуровневыми сумками.
- Для копирования вложенного контейнера доступен выбор конкретного контейнера внутри целевой укладки и слоты внутри верхнеуровневых сумок.
- Для вложенных контейнеров в picker показываются слоты вставки между соседними вложенными контейнерами; выбранный слот влияет на порядок.
- Для вещей picker выбирает конкретный контейнер; вещь не копируется молча "вне укладки" из шаблона.
- Кнопка "Скопировать всю укладку" остается отдельным сценарием и создает укладку из шаблона целиком.
