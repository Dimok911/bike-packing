# Механики копирования

Документ фиксирует ожидаемое поведение копирования сумок, вложенных сумок и
вещей. Если меняется UI или контракт копирования, сначала обновить этот файл,
затем код и регрессионные проверки.

## Термины

- **Пользовательская укладка** - приватная укладка гостя или авторизованного пользователя.
- **Demo-укладка** - основной опубликованный шаблон для выбранного языка. Без
  логина demo может стать стартовой локальной пользовательской укладкой, но
  опубликованная demo-укладка не становится личной укладкой пользователя сама
  по себе.
- **Shared-укладка** - дополнительная опубликованная укладка-шаблон из
  публичного каталога. Обычные пользователи видят ее как read-only источник для
  копирования, админ может открыть ее как редактируемый draft.
- **Шаблон** - собирательный термин для опубликованной demo-укладки или
  shared-укладки. В админском режиме шаблон может быть открыт как
  редактируемый draft.
- **Текущая укладка** - укладка, из которой пользователь начал действие. При
  копировании из шаблона текущий шаблон не должен быть целевой укладкой.
- **Место назначения** - конкретный контейнер внутри целевой укладки или корень целевой укладки.
- **Пространство** - изолированная область владения данными со своими
  укладками, каталогами вещей и сумок, справочниками, настройками и фото.
  Записи одного пространства можно переиспользовать по `id`; при копировании
  через границу пространств создаются записи, принадлежащие целевому
  пространству.
- **Личные данные пользователя** - все пользовательские укладки одного гостя
  или аккаунта. У них общий каталог вещей, общий каталог сумок, общие
  справочники и общие фото.
- **Собственные данные шаблона** - вещи, сумки, справочники, размещение и фото,
  которые принадлежат только конкретной demo/shared укладке. RU и EN версии
  одного шаблона считаются разными шаблонами со своими собственными данными.
- **Настройки пространства** - предметные настройки данных: категории, места и
  layout-поля. Локальные UI-предпочтения устройства (режим карточек, фильтры,
  сворачивание, visual style) не копируются как часть пространства.

## Модель пространств

- Все пользовательские укладки одного гостя или одного аккаунта принадлежат
  одному общему пользовательскому пространству. Они используют общие каталоги
  вещей и сумок, справочники, настройки пространства и фото.
- У гостя пользовательское пространство хранится локально на устройстве. У
  авторизованного пользователя оно хранится на сервере и синхронизируется с
  устройством. На правила копирования это различие не влияет.
- Каждый demo/shared-шаблон является отдельным пространством со своими
  укладкой, каталогами, справочниками, настройками и фото. RU- и EN-версии
  считаются разными шаблонами и разными пространствами.
- Draft-шаблон в админском режиме остаётся пространством соответствующего
  шаблона и не становится частью пользовательского пространства админа.
- Внутри одного пространства копирование может переиспользовать существующие
  записи по `id`. При пересечении границы пространств создаются изолированные
  записи целевого пространства, а исходное пространство не изменяется.

## Гость и обычный пользователь

Гость без логина и обычный авторизованный пользователь работают с одной
пользовательской механикой копирования. Отличается только хранение и
синхронизация их личных данных. Копирование из шаблона не должно менять сам
шаблон.

### Направления копирования

Гостю и обычному пользователю доступны только два направления копирования
отдельных вещей, сумок и деревьев сумок:

- **Пользовательская укладка -> пользовательская укладка.** Копирование
  происходит внутри одного пользовательского пространства. Существующие
  записи переиспользуются по `id`, а настоящий дубль создаётся только после
  явного выбора пользователя.
- **Шаблон -> пользовательская укладка.** В пользовательском пространстве
  создаются собственные записи. Шаблон-источник не изменяется.

Гость и обычный пользователь не могут копировать из пользовательской укладки
в шаблон или из одного шаблона в другой. Эти направления доступны только
админу в режиме редактирования шаблонов.

### Верхнеуровневая сумка

- В текущей укладке открывается root-level интерфейс со слотами между сумками.
- Для другой укладки сначала выбирается укладка, затем позиция сумки в ней.
- Из шаблона доступны только пользовательские укладки текущего гостя или
  аккаунта, которые ещё не содержат эту сумку.

### Вложенная сумка

- Всегда открывается выбор места внутри целевой укладки.
- Нельзя выбрать исходную сумку или одного из ее потомков.
- Из шаблона целью может быть только пользовательская укладка текущего гостя
  или аккаунта.

### Вещь

- Открывается выбор конкретной сумки или места в целевой укладке.
- Между пользовательскими укладками используется существующий `id` вещи.
- Настоящий дубль создается только отдельной кнопкой дублирования на карточке.
- Копирование из шаблона создаёт запись в личных данных пользователя и не
  меняет шаблон.

### Различия хранения и синхронизации

- Гость работает с локальными пользовательскими укладками. Если подходящей
  укладки ещё нет, при копировании из шаблона создаётся новая локальная
  укладка. Фото из шаблона кэшируются локально как fallback для последующей
  синхронизации после входа.
- Обычный пользователь работает с приватными серверными укладками.
  Скопированные из шаблона записи создаются в его private namespace и
  сохраняются на сервере.

Если исходный объект уже является копией из public/template, при копировании
сохраняется защита от перезаписи публичного происхождения и проверяются дубли.

## Проверка дублей внутри личных данных пользователя

Внутри пользовательских укладок одного пользователя копирование в другую укладку
работает как размещение существующих записей каталога, а не как создание новых
записей во вкладках "Вещи" и "Сумки".

- Вещь, которой еще нет в целевой укладке, добавляется в выбранную сумку по
  существующему `id`.
- Если эта вещь уже есть в целевой укладке, приложение должно спросить
  пользователя. Только явное действие "Дублировать" создает новую запись вещи в
  каталоге; отмена/пропуск не меняет каталог.
- Верхнеуровневая сумка, которой еще нет в целевой укладке, добавляется в
  целевую укладку по существующему `id`. Название не получает суффикс "копия",
  а вкладка "Сумки" не раздувается новой записью.
- Если такая сумка или часть ее дерева уже есть в целевой укладке, сначала
  выполняется проверка дублей. Сценарий "Только недостающие" восстанавливает
  отсутствующие вложенные контейнеры/пакеты и вещи по существующим `id` без
  создания новых записей каталога и без дублирования уже существующих элементов.
  Сценарий "Дублировать" создает настоящий дубль только после явного выбора
  пользователя.
- Специальные кнопки дублирования на карточках во вкладках "Вещи" и "Сумки" -
  отдельный сценарий. Они действительно создают новую запись каталога и могут
  добавить суффикс "копия".

## Модель копирования фото

Фото не являются самостоятельным глобальным справочником. Их область хранения
определяется тем, где находится укладка: в личных данных пользователя или в
собственных данных конкретного шаблона.

- **Пользовательские укладки одного пользователя** живут в одних личных данных.
  При копировании вещей, сумок и содержимого сумок между такими
  укладками фото физически не копируются: целевая укладка получает ссылку на
  тот же `id` вещи или сумки, а фото остаются ссылками на те же серверные файлы
  этого же списка. Если пользователь явно создает дубль специальной кнопкой на
  карточке, тогда появляется новая запись каталога.
- **Вложенный пакет/список внутри сумки** является частью конкретной укладки, а
  не общей root-level сумкой каталога. При переносе такого пакета в другую
  пользовательскую укладку создается новая вложенная структура целевой укладки,
  чтобы она жила независимо от исходной. При этом фото внутри тех же личных
  данных не отправляются через `/photos/copy`: если у фото уже есть
  серверные `url`/`thumbUrl`, новая вложенная структура сохраняет эти ссылки как
  `synced`; если фото еще только локальное, копируется локальный blob в
  IndexedDB под новым `localId`.
- **Шаблон** хранит собственные данные отдельно от личных данных пользователя.
  Каждый demo/shared/template list,
  включая отдельные RU/EN шаблоны, хранит собственный каталог и собственные
  файлы фото.
- В админском browser state записи draft-шаблона могут физически лежать рядом с
  приватными записями, но `publicCatalogLayoutId` обозначает их отдельного
  владельца. Такие записи нельзя link-by-ID подключать к пользовательской
  укладке; при пересечении границы всегда создается изолированная запись.
- Если копирование пересекает границу пользовательская укладка -> шаблон,
  шаблон -> пользовательская укладка или шаблон -> другой шаблон, фото должны
  быть физически скопированы в целевой list.
- Если server-side копирование фото нового шаблона переходит в upload fallback,
  техническая priming-запись целевого list должна оставаться `private`. Шаблон
  становится публичным только после успешной загрузки фото и финального save;
  гость не должен видеть промежуточный payload без изображений.
- Server-side ответ с целевыми URL фото нужно применять ко всему каталогу
  конкретного шаблона, включая вещи и сумки, уже удалённые из arrangement, но
  сохранённые как detached catalog records. Иначе следующая правка шаблона
  ошибочно повторит копирование по старым source URL.
- Маркер `_copyToCurrentList` допустим только для перехода через такую границу.
  Для копий внутри приватных пользовательских укладок этот маркер ставить
  нельзя, иначе приложение начнет дергать `/photos/copy` для фото, которые уже
  доступны в тех же личных данных пользователя.
- Явное private -> private дублирование через copy-to-layout может создать новую
  запись вещи или контейнера только после подтверждения пользователя, но оно все
  равно остается внутри тех же личных данных: remote photo id не меняются и
  `/photos/copy` не вызывается.

## Админ

Админ может копировать в пользовательские укладки и в другие редактируемые
draft-шаблоны. Текущий draft-шаблон не должен быть целевой укладкой при
копировании из него.

### Направления копирования

Эти направления относятся к копированию отдельных вещей, сумок и деревьев
сумок. Копирование укладки целиком является отдельным сценарием.

- **Пользовательская укладка -> пользовательская укладка.** Копирование
  происходит внутри одного пользовательского пространства. Существующие
  записи переиспользуются по `id`, а настоящий дубль создаётся только после
  явного выбора пользователя.
- **Пользовательская укладка -> draft-шаблон.** В пространстве целевого
  шаблона создаются собственные записи. Личный каталог пользователя и
  исходная укладка не изменяются.
- **Draft-шаблон -> пользовательская укладка.** В пользовательском
  пространстве создаются собственные записи. Шаблон-источник не изменяется.
- **Draft-шаблон -> другой draft-шаблон.** В пространстве целевого шаблона
  создаются собственные записи. Исходный draft нельзя выбрать целью.

### Верхнеуровневая сумка

- Для пользовательской укладки открывается root-level интерфейс со слотами.
- Из draft-шаблона целями могут быть пользовательские укладки и другие
  draft-шаблоны; сам исходный draft из списка исключается.

### Вложенная сумка

- Открывается выбор конкретного места в пользовательской укладке или другом
  draft-шаблоне.
- Нельзя выбрать исходную сумку или одного из её потомков.

### Вещь

- Для пользовательской цели внутри private namespace используется существующий
  `id`; отдельный дубль создается только явным действием.
- Для draft-шаблона создается новая запись, принадлежащая namespace целевого
  шаблона.
- Из шаблона можно копировать в пользовательскую укладку или другой draft, но
  не в тот же исходный draft.

При копировании вещи или сумки в draft-шаблон в пространстве целевого шаблона
создаётся собственная запись. Она не становится частью личного каталога
админа. При копировании в пользовательскую укладку запись принадлежит
пользовательскому пространству; сведения о шаблоне-источнике могут сохраняться
только для проверки дублей и не делают запись частью пространства шаблона.

## Public/template metadata

- A new shared template id must keep `public-shared-layout-${sharedId}` within the API's 64-character list-id boundary. A save must also adopt the canonical `listId` returned by the API before checking or uploading photos.

Серверный список опубликованных demo/shared/template записей является
авторитетным источником того, какие шаблоны видны в админке и без логина.
Фронт не должен показывать template-строку только потому, что она осталась в
локальном state или runtime-каталоге.

- Копия опубликованного шаблона считается видимой только после подтверждения
  сервера и повторного получения серверного каталога.
- Demo и референс-шаблоны используют одну metadata-модель: роль, `id`,
  название, язык, видимость, тип источника и время обновления живут в
  серверной строке `bike_packing_lists`.
- Язык шаблона хранится как metadata (`bike_packing_lists.language`) и не
  переводит содержимое payload.
- RU и EN версии одного шаблона - разные опубликованные записи с разными
  собственными данными. Для полноценной пары нужны две серверные версии.
- Если серверной записи шаблона нет, фронт не восстанавливает ее из старого
  встроенного набора или локальных следов.
- При смене языка публичного шаблона фронт выбирает шаблон из списка нового
  языка, а не подменяет metadata поверх payload старого языка.
- Без логина demo может стать стартовой локальной пользовательской укладкой.
  Повторный вход не должен плодить новые demo-копии без явного действия
  пользователя.
- Удаление опубликованного шаблона сначала подтверждается сервером. После
  удаления фронт очищает локальные следы этой записи и остается в зоне
  шаблонов/demo, а не переключается в пользовательские укладки.

## UI-контракт

- Индивидуальная кнопка копирования сумки или вещи из шаблона, включая кнопку внутри окна редактирования в админке, открывает picker места назначения. При выборе личной укладки всегда создаётся независимая личная запись, а исходная shared-запись не привязывается по ID.
- Копирование сумки из вкладки «Сумки» создаёт пустую сумку: сохраняются свойства и фото самой сумки, но не копируются вложенные сумки и вещи. Копирование дерева вместе с содержимым выполняется из укладки.
- Для личной копии вещи или сумки статус синхронизации очищается только после подтверждения сервером новой записи и изменённой целевой укладки; отсутствие ожидаемого `upserted` запускает полный save fallback.
- Picker в режиме копирования показывает выбор целевой укладки, если доступно больше одной цели.
- Для копирования верхнеуровневого контейнера используется root-level доска со слотами между верхнеуровневыми сумками.
- Имя новой верхнеуровневой сумки проверяется только внутри целевой укладки. Если такого имени там нет, оно сохраняется без суффикса; при конфликте используется локализованный суффикс `копия`/`copy`.
- Для копирования вложенного контейнера доступен выбор конкретного контейнера
  внутри целевой укладки и слоты внутри верхнеуровневых сумок.
- Для вложенных контейнеров в picker показываются слоты вставки между соседними
  вложенными контейнерами; выбранный слот влияет на порядок.
- Для вещей picker выбирает конкретный контейнер; вещь не копируется молча "вне укладки" из шаблона.
- Кнопка "Скопировать всю укладку" остается отдельным сценарием и создает укладку из шаблона целиком.
