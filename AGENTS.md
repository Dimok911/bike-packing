# Правила проекта

## Кэш фронта и сборка

- При изменениях фронтового runtime-кода, UI, стилей, service worker или любых файлов, которые должны быть видны в браузере, обновлять все точки версии/кэша вместе:
  - `src/config/constants.js` -> `APP_VERSION`
  - корневой `index.html` -> query string для `styles.css?v=...` и `app.js?v=...`
  - корневой `sw.js` -> `CACHE_NAME` и query string в `ASSETS`
- Нельзя обновлять только `APP_VERSION` или только `sw.js`: на `localhost` браузер загружает именно те URL, которые прописаны в корневом `index.html`.
- После таких изменений запускать:
  - `npm.cmd run check`
  - `npm.cmd run build`
- Перед тем как просить пользователя проверить в браузере, убедиться, что старых ссылок на версию не осталось. Пример:
  - `rg -n "v622|v623|v624|app\\.js\\?v=|styles\\.css\\?v=" index.html sw.js src/config/constants.js`
  Номера версий в поиске менять под текущий bump.
- Если браузер всё ещё показывает старую версию, сначала подозревать старую вкладку, service worker или уже запущенный dev-server со старыми файлами. Сначала проверить реальные файлы, потом уже лезть в бизнес-логику.

## Модульность и `app.js`

- `app.js` считать точкой сборки сценариев, а не местом для бесконечного накопления бизнес-логики. При редактировании общего `app.js` нужно смотреть, есть ли рядом повторяющиеся или смысловые функции, которые можно вынести в модуль.
- Если новая функция относится к отдельному смыслу, по умолчанию добавлять её в подходящий модуль в `src/...`, а в `app.js` оставлять только orchestration: вызовы, связывание с UI, состояние текущего экрана.
- ВАЖНО! При любой правке `app.js` выносить в модули не только новый код, но и всё смысловое/повторяющееся, с чем пришлось соприкоснуться во время редактирования. Не добавлять новые helper-функции в `app.js`, если их можно безопасно положить в `src/state`, `src/ui`, `src/sync`, `src/public` или другой подходящий домен.
- Если при правке приходится трогать 2-3 похожих helper-функции в `app.js`, это сигнал вынести их вместе. Не оставлять рядом несколько вариантов одной идеи с разными названиями.
- Вынесение делать небольшими безопасными шагами: сначала перенести чистые функции без доступа к DOM/global state, затем функции с зависимостями через явные параметры, и только потом упрощать вызывающий код.
- Не смешивать в одном модуле разные уровни ответственности. Модуль должен отвечать за один домен: фото, shared-layout index, layout arrangement, удаление layout, API-клиент, UI-диалог и т.п.
- Новые модули обязательно подключать к `npm.cmd run check`: если файл добавлен в `src/...`, проверить, надо ли расширить script `check` в `package.json`.
- После выноса логики из `app.js` проверять не только сборку, но и регрессионный сценарий, ради которого правка делалась. Для шаблонов это минимум: копирование, фото, переключение вкладок/укладок, удаление.

Идеальная структура для быстрого редактирования и поиска багов:

- `app.js` - старт приложения, связывание модулей, обработчики UI, маршрутизация режимов, минимальная orchestration-логика.
- `src/config/` - константы, версии, endpoints, feature flags.
- `src/sync/` - API-клиент, синхронизация, фото, сериализация, история. Всё, что ходит в сеть или готовит payload.
- `src/state/` - чистая работа с состоянием: layout arrangement, нормализация, удаление, метрики, словари, имена, диагностика.
- `src/public/` - логика public/demo/shared: копирование публичного в приватное, shared-layout index, admin shared delete, guest demo, scope/read-only правила.
- `src/ui/` - рендер и поведение отдельных UI-частей: диалоги, форматирование, поиск, подтверждения, печать. UI-модуль не должен сам решать API-контракт.
- `src/data/` - встроенные demo/shared данные и справочники.
- `src/storage/` - localStorage/session/storage scope и низкоуровневое хранение.
- `src/utils/` - маленькие чистые утилиты без знания предметной области.
- `docs/` - журналы расследований и архитектурные заметки, особенно для сложных регрессий.

Для текущей больной зоны шаблонов желаемое разделение такое:

- `src/public/shared-layouts.js` - только runtime/index shared-шаблонов и нормализация index entry.
- `src/public/shared-layout-admin.js` - admin delete/publish-adjacent операции shared-шаблонов, без UI.
- `src/sync/photos.js` - распознавание, загрузка и копирование фото через API.
- `src/state/layout-manage.js` - признаки admin/demo/shared/template-copy layout и чистые операции создания record.
- `src/state/layout-delete.js` - удаление дерева layout из state.
- `app.js` - пользовательский сценарий: нажали "копировать", "сохранить", "удалить", показать статус/тост, вызвать нужные модули.

## Где лежит API

- Репозиторий API/backend для bike-packing лежит здесь:
  - `C:\Users\user\Documents\repo\bike-packing\bikepacking-api`
- Изменения фронта в этом репозитории и изменения API-контракта/backend в `bikepacking-api` нужно проверять, коммитить и пушить отдельно, если пользователь явно не попросил только одну сторону.
- Если `bikepacking-api` обновлялся, нельзя оставлять изменения просто локально. Нужно проверить `git status`, запустить доступные проверки/тесты API, сделать commit, push и явно написать пользователю, что API закоммичен и запушен.
- При отладке копирования шаблонов, копирования фото, удаления shared-шаблонов или public demo/shared index смотреть и фронт, и `bikepacking-api`, прежде чем делать выводы о контракте.

## API compatibility и предупреждения админки

- Любое изменение backend/API, которое влияет на поведение фронта, админские операции, CORS, HTTP-method, payload, валидацию, фото, demo/shared/template endpoints или диагностику совместимости, должно сопровождаться bump `bikePackingApiCompatibilityVersion` в `bikepacking-api/server.mjs`.
- Если фронт должен заранее понять, что API еще не обновлен или процесс backend не перезапущен, одного изменения кода API недостаточно: нужно добавить/обновить capability в `bikePackingApiCapabilities` и потребовать ее во фронте в `REQUIRED_ADMIN_API_CAPABILITIES`.
- При изменении frontend-кода, который зависит от нового API-поведения, синхронно обновлять `REQUIRED_ADMIN_API_VERSION` в `app.js`. Иначе старый запущенный API может честно отдать прежнюю версию и фронт не покажет предупреждение.
- Нельзя менять API-поведение и оставлять прежний compatibility version только потому, что endpoint "почти тот же". Если пользователь должен увидеть предупреждение до перезапуска/деплоя API, версия или capability обязаны измениться.
- После таких правок проверять обе стороны: frontend `npm.cmd run check` + `npm.cmd run build`, backend `npm.cmd run check`, затем коммитить и пушить frontend/backend отдельно.
